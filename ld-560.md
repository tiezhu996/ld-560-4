## 项目概述

**交通出行实时监控大屏（TransitDashboard）** — 面向城市交通管理部门的实时监控可视化大屏系统。通过 WebSocket 模拟数据流，展示车辆实时位置、充电桩状态、路线运行状况和订单分析，支持高德地图集成实现地理空间数据可视化。

**项目类型：数据可视化大屏（纯前端）**

## 项目主要需求

### 一、核心数据模型

1. **车辆实时状态 VehicleLive**：字段包括 id、plateNumber、type（公交/出租/网约车/物流）、lat、lng、speed、direction、status（行驶中/停靠/故障/空载）、lastUpdate。业务动作：实时位置更新、状态变更通知。此模型贯穿全栈：类型定义 → 状态管理 → WebSocket 接收 → 地图标记组件，主屏和车辆监控屏均消费此模型。

2. **充电桩状态 ChargingPileLive**：字段包括 id、code、lat、lng、type（快充/慢充）、status（空闲/充电中/故障/维护）、currentPower、todayUsage、queueCount。业务动作：状态实时监控、故障告警推送。此模型贯穿全栈：类型定义 → 状态管理 → WebSocket 接收 → 地图标记 + 状态图表，主屏和充电桩监控屏均消费此模型。

3. **出行订单统计 OrderStats**：字段包括 date、totalOrders、completedRate、avgDistance、avgDuration、revenue、byType（按车型分类的子统计）。业务动作：实时统计刷新、趋势分析。此模型贯穿全栈：类型定义 → 状态管理 → Mock 生成器 → 图表组件，主屏和订单分析屏均消费此模型。

4. **路线监控 RouteMonitor**：字段包括 routeId、name、vehicles（当前在线车辆列表）、avgSpeed、passengerLoad、delayMinutes。业务动作：路线运行状况监控、准点率计算。此模型贯穿全栈：类型定义 → 状态管理 → Mock 生成器 → 图表组件，路线监控屏消费此模型。

### 二、核心页面（≥ 4 个）

1. **`/` 主屏**：三栏布局。左侧区域显示车辆实时统计面板（各类型在线数、故障数）+ 充电桩状态面板（空闲/充电中/故障占比），中间为高德地图主视图（聚合展示车辆标记和充电桩标记，支持缩放和点击弹窗），右侧显示订单趋势折线图 + 收入数据卡片。消费 VehicleLive + ChargingPileLive + OrderStats 全部模型。

2. **`/vehicles` 车辆监控屏**：上方为各类型车辆实时位置分布的高德地图（可按类型筛选标记），下方左侧为车辆状态分布饼图，下方右侧为在线率趋势折线图（24小时维度）。消费 VehicleLive 模型。

3. **`/charging` 充电桩监控屏**：充电桩利用率仪表盘（ECharts gauge）、排队趋势折线图、故障告警列表（实时滚动）、今日收入统计卡片 + 收入趋势面积图。消费 ChargingPileLive 模型。

4. **`/routes` 路线监控屏**：左侧为路线列表（可选中查看），中间为选中路线的高德地图路径展示（途经站点标记），右侧为准点率仪表盘 + 客流热力图（ECharts heatmap，按小时 x 路线维度）。消费 RouteMonitor 模型。

5. **`/orders` 订单分析屏**：实时订单流水滚动列表、完成率仪表盘（ECharts gauge）、订单类型分布玫瑰图、日/周/月趋势切换折线图。消费 OrderStats 模型。

### 三、共用组件

- `VehicleMap`：车辆地图标记组件，展示车辆实时位置（被主屏、车辆监控屏使用）
- `ChargingStatusChart`：充电桩状态图表（被主屏、充电桩监控屏使用）
- `OrderTrendChart`：订单趋势折线图（被主屏、订单分析屏使用）
- `PassengerHeatMap`：客流热力图（被路线监控屏使用）
- `SpeedGauge`：速度仪表盘（被车辆监控屏、路线监控屏使用）
- `DataCard`：数据卡片组件（被所有页面使用）
- `AlertBanner`：告警横幅组件（被充电桩监控屏、主屏使用）
- `AnimatedNumber`：数字动画组件（被所有数据卡片使用）
- `ScreenAdapter`：大屏自适应容器（被所有页面使用）
- `MapContainer`：高德地图容器组件（被所有含地图的页面使用）

### 四、横切关注点

1. **WebSocket 实时数据流**：模拟车辆位置更新（每秒刷新坐标）和充电桩状态变更（每 5 秒更新状态）。Mock WebSocket Server 在前端进程中运行，使用 `useWebSocket()` 组合式函数统一管理连接生命周期（建立、重连、断开），全局状态通过 Pinia Store 接收和分发数据。此关注点触达 WebSocket mock 模块、全局状态管理、所有图表组件的自动刷新。

2. **大屏适配（1920x1080 基准）**：全局使用 rem/vw 适配方案 + ECharts 自适应 resize。ScreenAdapter 组件监听窗口变化，统一计算缩放比例。此关注点触达全局样式和每个图表组件。

## 代码架构要求（强制执行）

### 枚举必须单独维护为常量文件

所有枚举类型必须在 `src/constants/enums.ts` 中集中定义，业务代码中**严禁硬编码**枚举值字符串。README 中必须列出所有枚举的定义位置和使用位置对照表。

**共享枚举定义**：
- `VehicleType`：BUS / TAXI / RIDE_HAILING / LOGISTICS
- `VehicleStatus`：RUNNING / PARKING / FAULTY / IDLE
- `PileType`：FAST / SLOW
- `PileStatus`：IDLE / CHARGING / FAULTY / MAINTENANCE

### 全局异常处理

前端请求拦截器统一错误提示：在 `src/utils/request.ts` 中封装 Axios 实例，配置响应拦截器拦截 HTTP 错误和业务错误码，统一调用错误提示组件展示，禁止各页面自行 catch 后直接 alert。WebSocket 连接错误通过 `useWebSocket()` 中统一处理重连和错误提示。

### 操作日志

关键操作（WebSocket 连接状态变更、告警确认、数据导出）通过审计工具函数记录操作日志。在 `src/utils/audit-log.ts` 中封装，记录操作类型、操作时间、操作详情。日志存储在内存队列中，支持按时间导出。

### 代码严禁合并职责到单一文件

- 每个功能模块必须拆分为独立文件。例如：车辆模块拆分为 `vehicleStore.ts`（状态管理）、`vehicle.d.ts`（类型定义）、`VehicleMap.vue`（地图组件）、`useVehicleData.ts`（组合式函数）、`vehicleMock.ts`（数据生成器）。
- 鼓励细粒度拆分，单个文件不超过 300 行。
- 前端组件按功能拆分，每个页面拆分为多个子组件。
- 按实体/功能拆分为 store、types、constants、components、hooks、mock、utils 等独立文件。

### 核心实体贯穿全栈

每个核心实体必须完整贯穿：**类型定义（types/*.d.ts）→ 状态管理（stores/*.ts）→ 组合式函数（hooks/*.ts）→ Mock 生成器（mock/*.ts）→ 页面组件（pages/*.vue）→ 图表/地图组件（components/charts/*.vue）**。禁止在页面中直接操作原始数据，必须通过 store 中转。

## 部署要求

项目为纯前端数据可视化大屏，无需后端服务和数据库。所有数据通过 Mock WebSocket 模拟实时推送。

```bash
npm run dev
```

访问 http://localhost:38409 查看大屏。

**高德地图 API Key 配置**：在项目根目录创建 `.env` 文件，写入：

```
VITE_AMAP_KEY=your_amap_key_here
```

Vite 会通过 `import.meta.env.VITE_AMAP_KEY` 注入到运行时。`.env` 文件不得提交到 Git。
